## 한 줄 요약

곰곰이는 Slack에서 자연어로 데이터 질문을 받으면, ReAct 기반 AI 에이전트가 기억 저장소와 지식 저장소를 먼저 조회한 뒤 적절한 쿼리 엔진을 선택해 SQL을 실행하고, 필요하면 CSV까지 제공하는 사내 데이터 분석 AI 에이전트다.

## 왜 만들었는가

- 기존 BI 대시보드는 정해진 화면 안에서만 데이터를 볼 수 있어 새로운 질문에 유연하게 대응하기 어렵다.
- 데이터 확인이 개발자에게 집중되면 기획, 마케팅, 운영 조직의 데이터 접근성이 떨어지고 소통 비용이 커진다.
- BigQuery, Athena, PostgreSQL, SaaS 등으로 정보가 흩어져 있어 통합적인 확인이 어렵다.
- 이상 징후를 사후에 확인하는 방식에서 벗어나, 더 빠른 분석과 선제 대응이 가능한 환경이 필요했다.

## 곰곰이의 핵심 아이디어

- 단순한 "자연어 -> SQL 변환기"가 아니라, 여러 도구를 자율적으로 선택하는 AI 에이전트로 설계했다.
- 질문을 받으면 바로 SQL을 만들지 않고, 먼저 과거 기억과 참고 지식을 검색해서 정확도를 높인다.
- 잘 해결된 대화에서 도메인 지식, 분석 절차, 사용 SQL을 추출해 장기 기억으로 저장한다.
- 이후 유사한 질문이 들어오면 과거 기억을 재활용해 더 빠르고 안정적으로 답한다.

## 사용된 주요 도구

### 1. SQL 실행 도구

- 데이터베이스 조회와 분석에 사용한다.
- 토큰 낭비를 막기 위해 한 번에 조회하는 결과를 제한한다.

### 2. CSV 변환 도구

- 결과가 많을 때 CSV 파일로 변환한다.
- 사용자는 다운로드 링크로 대용량 데이터를 받는다.

### 3. 예시 SQL/지식 저장소

- 테이블 스키마, 참고용 SQL, 해결 사례를 저장한다.
- SQL 생성 전에 관련 지식을 불러와 품질을 높인다.

### 4. 장기 기억 저장소

- 이전 성공 사례, 분석 순서, 사용자가 알려준 도메인 지식을 저장한다.
- Vector DB, Embedding, Semantic Search 기반으로 유사 기억을 찾는다.

## 에이전트 동작 방식

아티클에서 설명하는 곰곰이의 핵심 흐름은 다음과 같다.

1. 사용자가 Slack에서 질문한다.
2. API 서버를 거쳐 AWS Lambda 상의 LangGraph 기반 에이전트가 실행된다.
3. 에이전트는 ReAct 방식으로 추론하고, 필요한 도구를 순서대로 선택한다.
4. 매 질문마다 먼저 Memory Tool로 유사한 기억을 찾는다.
5. Query Tool을 쓰기 전 Knowledge Base Tool로 스키마와 예시 SQL을 확인한다.
6. 필요하면 BigQuery 또는 Athena 중 적합한 쿼리 엔진을 선택한다.
7. 결과가 크면 CSV Download Tool로 파일 링크를 만든다.
8. 잘 해결된 대화는 다시 기억 저장소에 기록해 다음 질문에 활용한다.

## 시스템 구조 요약

- UI: Slack
- API 계층: 요청 수신, 에이전트 실행 연결
- Agent 계층: LangGraph + ReAct 프롬프트
- Tool 계층: Memory, Knowledge Base, Query, CSV Download
- Data 계층: BigQuery, Athena, PostgreSQL
- AI/검색 계층: Embedding Model, Semantic Search, Vector DB

특히 기억 저장소는 Aurora Serverless v2 PostgreSQL과 `pg_vector`를 활용해, 사용량이 적을 때 비용 효율적으로 운영되도록 설계했다.

## 도입 후 효과

- 동료들이 개발자를 거치지 않고 직접 데이터를 확인할 수 있게 되었다.
- 반복적인 SQL 요청과 소모적인 질의응답이 크게 줄었다.
- AI가 "계획 수립 -> SQL 작성 및 실행 -> 인사이트 도출" 흐름을 빠르게 수행하게 되었다.
- 조직의 데이터 활용 능력이 올라가고, 개발자는 반복 업무 대신 더 본질적인 엔지니어링에 집중할 수 있게 되었다.

## 글에서 강조하는 시사점

- AI 에이전트의 성능은 모델 자체보다 "어떤 도구를 주고, 어떤 순서로 쓰게 하느냐"에 크게 좌우된다.
- 데이터 분석 에이전트는 단발성 답변보다 "기억하고 재사용하는 구조"가 중요하다.
- 정적인 BI 도구를 완전히 없애기보다는, AI 에이전트와 상호 보완적으로 운영하는 것이 현실적이다.
- 다음 단계는 능동적인 이상 탐지, 예측, 웹 기반 시각화, 멀티 에이전트 확장이다.

## 구현 관점에서 뽑을 수 있는 핵심 요구사항

- Slack 기반 자연어 질의 인터페이스
- ReAct 또는 동등한 도구 호출형 에이전트 루프
- 스키마/예시 SQL/용어집 검색 기능
- 장기 기억 저장 및 유사 질의 회수 기능
- 다중 쿼리 엔진 선택 로직
- 안전한 read-only SQL 실행 제어
- 대용량 결과 CSV 생성 및 다운로드 링크 제공
- 성공/실패 사례를 다시 학습 자산으로 축적하는 피드백 루프

## 결론

이 글에서의 곰곰이는 단순한 챗봇이 아니라, 조직 내 데이터 접근 병목을 줄이기 위해 설계된 "기억하는 SQL 분석 에이전트"다. 핵심 경쟁력은 자연어 이해 자체보다도, 질문마다 기억과 지식을 먼저 검색하고 적절한 도구를 조합해 문제를 해결한 뒤, 그 경험을 다시 축적하는 구조에 있다.
